> 对象：plan-claude.md / plan-codex.md / plan-gemini.md
> 参照：src/index.html（当前上线版本）

## 一、逐方案点评

### 1. Claude 方案 ——「年卡护照」

**优点**

- **问题诊断最锐利**。P1-P8 的分级排序是三份方案里唯一做了严重度标注的，说明作者真正用过这个页面。特别是 P2（回本后体验断崖）和 P6（战报卡片重复感）直接命中当前版本最大的两个情感空洞。
- **隐喻升级逻辑自洽**。"票根是一次性的，护照是积累型的"——这句话一下子解释了为什么需要改，方向在哪。从单次凭证到长期收藏品，隐喻本身就是产品策略。
- **成就墙是三份方案中最有价值的单一想法**。3×3 印章收集网格完美解决了"回本后没有追踪动力"的问题，同时给了用户盖章的仪式感——这比进度条强十倍。
- **分期实施有优先级**。Phase 1/2/3 的拆分说明作者考虑了工程成本，不是在画大饼。

**缺点**

- **想法太多，刹不住车**。MRZ 机读区、可扫描 Code128 条形码、html2canvas 多级截图、蜡封印章、烫金边框……全做完不是护照，是过度设计的工艺品。一个单 HTML 文件装不下这么多想法。
- **护照隐喻有过度延伸的风险**。"签证页""MRZ""出入境记录"——这些词在环球影城的语境里是冷的。游乐园是关于快乐和消费的，不是关于出入境管理的。隐喻走到后半段已经开始为了护照而护照。
- **叠卡交互（iOS 钱包式）描述得漂亮但工程量被低估**。拖拽展开 + 弹性回弹 + z-index 层叠 + 滚动容器嵌套，在单 HTML 里要写出流畅体验，JS 量会爆炸。
- **双半环图对非设计师用户没有认知优势**。左收入右支出，看起来不如简单的列表直觉。环形图在移动端小屏下更是灾难——弧段太短的分类会变成一根线。

### 2. Codex 方案 ——「档案馆票证 / Archive Pass」

**优点**

- **三份方案中最克制、最成熟**。"克制但有纪念价值"六个字就把调性定死了。不是加东西，而是减东西——降低纹理噪声、收敛配色到 1 主 2 辅、分类色从高饱和改为明度层级。这是唯一一份在做减法的方案。
- **信息架构思考最深**。5.1 的"单屏优先级"和 5.2 的文案语气建议（"结论先行 + 证据补充"）是三份方案中唯一在认真讨论阅读体验的。"首屏 3 秒内可回答'是否回本、差多少'"——这是设计验收指标，不是模糊的审美判断。
- **时间轴合并 season + mini-tickets 是最优雅的结构简化**。把两个功能相似但视觉割裂的区块合成一个纵向时间轴，信息密度不降、区块数减一、视觉节奏改善，一举三得。
- **"战报点击进抽屉而非铺在页面"**——这是正确的交互模式。当前版本的明细全铺开确实让页面变成了无限长面条。

**缺点**

- **太保守了，缺少一个让人兴奋的"大想法"**。通篇读下来感觉是在做 1.5 版，不是 2.0。"降噪、收敛、精简"都对，但没有一个亮点能让用户说"哇"。对比 Claude 的成就墙或 Gemini 的勋章动效，Codex 方案缺少情绪锚点。
- **"档案馆"调性可能矫枉过正**。当前版本的问题不是"太热闹"，而是"热闹得没有层次"。把气质从快乐庆典拉到博物馆档案室，是从一个极端跳到了另一个极端。环球影城终归是个游乐园，冷峻克制不应该是主调。
- **甜甜圈图在窄屏的降级方案（"仅保留排名表 + 进度条"）等于承认了甜甜圈在移动端站不住脚**。那为什么要先提甜甜圈？直接用排名表不就好了？
- **缺少成就/激励系统**。回本后用户凭什么继续看？方案只说"未完成态也给阶段奖章"，但没有展开。这是当前版本最大的痛点之一，Codex 基本绕过了。
- **"导出为海报 / 打印 A6"**——想法好，但在单 HTML 中实现 print CSS 是一个巨大的坑，且打印出来纸纹/渐变/阴影全部失效，等于要维护两套视觉系统。

### 3. Gemini 方案 ——「荣耀勋章」

**优点**

- **社交传播意识最强**。"做成竖版全屏卡片，用户可以直接截图发朋友圈/小红书"——这是三份方案中唯一从传播终态倒推设计的。勋章造型天然适合做头像框 / 分享卡，比票根和档案都更适合社交场景。
- **情感张力最大**。RPG 属性卡、荣誉之路、战绩揭晓……方案用了游戏化的语言重新包装了数据，让"回本百分比"变成了"成就等级"。对于目标用户（愿意买年卡的主题乐园爱好者），这套语言是有吸引力的。
- **成就文案池（"初入江湖"→"传奇降临"）**是三份方案中最有传播性的文案设计。这些文案本身就是社交货币——"我是传奇降临，你呢？"

**缺点**

- **过度设计到了夸张的程度**。金属拉丝纹理、3D 浮雕、光芒扫过、黑胶唱片纹理、环形罗盘、蚀刻效果、粒子化虚线……这不是设计方案，这是特效清单。一个内联 CSS+JS 的 HTML 文件要实现这些效果，代码量会膨胀到不可维护。
- **从"票根"到"勋章"是隐喻断裂，不是升级**。当前版本的票根美学已经建立了用户认知，直接切换到金属勋章风格等于放弃了所有积累的品牌资产。Claude 的"票根→护照"是延伸，Codex 的"票根→档案卡"是收敛，Gemini 的"票根→奖牌"是推翻重来。
- **横向滑动相册在移动端是伪需求**。用户在一个数据查询页面里不期待横滑交互——这是内容消费 App（抖音、小红书）的模式。而且横滑让用户无法纵向对比不同入园的数据，反而降低了信息效率。
- **"环形罗盘"资金流向图完全不可读**。扇形角度 + 环形渐变 + 中心数字 + 悬停 tooltip——在 320px 宽的手机屏上，这些元素会挤成一坨。Pie/donut chart 在可视化领域本身就是被诟病的图表类型，加上罗盘造型更是雪上加霜。
- **动效节奏失控**。加载序列长达 1800ms+ 且分了 5 个阶段，用户在等骨架屏消失的时候已经失去耐心了。好的动效应该在数据已经可读之后才开始装饰性演出，而不是让装饰性演出延迟信息呈现。
- **"可定制刻字"（用户输入座右铭）**——数据从 Notion 注入 `__DATA__`，页面没有写入能力，这个功能在架构上就不成立。

## 二、每份方案最好的 3 个想法

|---|------|------|

## 三、在"单 HTML 文件内联 CSS+JS"里会翻车的想法

| 想法 | 来源 | 翻车原因 |
|------|------|---------|
| **html2canvas 多级截图** | Claude | 需要引入 ~40KB 外部库，在单 HTML 里要么 inline 巨量代码要么加 CDN 依赖；且 html2canvas 对 CSS 变量、SVG、渐变的渲染 bug 多到令人发指 |
| **可扫描 Code128 条形码** | Claude | 需要 barcode 生成库或手写编码算法，ROI 极低——谁会真的扫一张截图上的条形码？ |
| **iOS 钱包式叠卡拖拽** | Claude | 拖拽 + 弹性回弹 + z-index 层叠 + 嵌套滚动容器，JS 量 300+ 行起步，且在不同浏览器/设备上手感差异巨大，调试成本极高 |
| **横向滑动相册 + 视差** | Gemini | 需要 touch 事件处理、惯性滚动、snap 点计算、视差层偏移，代码量不低于一个小型 carousel 库，且和页面纵向滚动冲突 |
| **环形罗盘资金流向** | Gemini | SVG 扇形 + 环形渐变 + hover 交互 + 动态角度计算，在 320px 屏幕上不可读，代码复杂度远高于收益 |
| **3D 浮雕文字 + 金属拉丝纹理** | Gemini | 多层 text-shadow 叠加 + 金属渐变在低端安卓机上会导致严重的 paint 性能问题；且 CSS 量爆炸而视觉提升有限 |
| **骨架屏 → 揭晓动画序列** | Gemini | 数据是 inline 在 HTML 里的（`__DATA__`），不存在"加载中"状态。骨架屏 + 1800ms 揭晓序列是在人为制造等待，用户会觉得页面卡了 |
| **打印 A6 / 导出海报** | Codex | print CSS 要重写所有渐变/阴影/纹理的降级方案，等于维护两套视觉。且纸质打印丢失所有质感细节，效果必然大打折扣 |
| **用户可定制刻字** | Gemini | 页面无后端、无持久化，数据来自 Notion 注入。用户输入的内容无处保存，刷新即丢失 |

## 四、融合最优方案

### 总体策略

**保留票根灵魂，吸收护照的积累感，借鉴档案馆的克制，偷走勋章的社交文案。**

不做隐喻大换血（不叫护照也不叫勋章），继续叫"年卡回本战绩票"，但从结构和细节上吸收三份方案的精华。

### 区块架构

```
┌──────────────────────────────────┐
│  A. Hero 票根（保留 + 微调）       │  核心社交截图区
├──────────────────────────────────┤
│  B. 赛季概览条（合并重构）          │  取代 mini-tickets + season-card
├──────────────────────────────────┤
│  C. 成就印章墙（新增）             │  回本后持续追踪的核心驱动力
├──────────────────────────────────┤
│  D. 趋势图（SVG 重构）            │  从 Canvas 迁移到 SVG
├──────────────────────────────────┤
│  E. 入园战报（折叠重构）            │  按月分组 + 抽屉式明细
├──────────────────────────────────┤
│  F. 消费分类（简化重构）            │  分离式列表取代双向条形图
├──────────────────────────────────┤
│  G. 尾注（增强）                   │  成就称号 + 下一目标
└──────────────────────────────────┘
```

**比当前版本少了 1 个区块**（season-card 和 mini-tickets 合并），**多了 1 个区块**（成就墙），净增为 0。页面不会更长。

---

### A. Hero 票根

**来源**：Claude（回本后切净赚金额）+ Codex（精简动效）+ 保留现有核心

| 改什么 | 具体做法 |
|--------|---------|
| 回本后数字 | 百分比 → 净赚金额（`+¥582`），金色渐变，印章改 `PROFIT` |
| 印章档位 | 扩展为 6 档（采用 Claude 的 ROOKIE → LEGEND 体系，但文案换成 Gemini 风格的中文：「新手上路」→「传奇降临」） |
| 动效克制 | 采用 Codex 建议：首屏只保留 2 个动画——数字 countUp + 印章轻压，不做飞入/光芒扫过 |
| 视觉微调 | 纸纹降 10% 不透明度（Codex），保留穿孔/条形码/倾斜（现有），不做烫金边框（过度） |

**不做**：可扫描条形码、蜡封质感、3D 浮雕、金属拉丝——这些在单 HTML 里 ROI 太低。

---

### B. 赛季概览条

**来源**：Codex（时间轴合并思路）+ Claude（概览条布局）

将 mini-tickets（3 张小卡）和 season-card（进度条）**合并为一个区块**：

```
┌─────────────────────────────────┐
│  S1 · 2025.01 — 2026.01         │
│                                  │
│  ¥1,820 节省  ¥380 消费  ¥1,440 净值 │
│  · · · · · · · · · · · · · · ·  │
│  8 次入园 · 62 天 · ¥180/次      │
│                                  │
│  ●━━━━━━━━●━━━━━━━●━━━━━○        │
│  25%     50%    75%    100%      │
│  1.15    2.08   3.12    ----     │
└─────────────────────────────────┘
```

- 上半部分：核心数字一行排完（采用 Claude 的紧凑布局），不再用 3 张独立卡片
- 下半部分：极简进度条 + 里程碑节点（采用 Codex 的节点附带达成日期设计）
- 节点已达成为实心 + 日期，未达成为空心 + `----`
- 取消微旋转——概览条要的是信息效率，不是装饰

### C. 成就印章墙（新增区块）

**来源**：Claude（成就墙核心设计）+ Gemini（成就文案）

这是融合方案中**最重要的新增区块**，解决"回本后无追踪动力"。

```
┌─────────────────────────────────┐
│  STAMPS   [4/9]                  │
│                                  │
│  [🎫 首次入园] [⭐ 五次常客] [👑 十次元老] │
│   2025.01.15    2025.03.08    ----    │
│                                  │
│  [🏆 回本成功] [💰 赚回五百] [🔥 单日之王] │
│   2025.04.22    ----         2025.02.14 │
│                                  │
│  [⚡ 周末战士] [🎪 全勤选手] [🌙 夜场达人] │
│   ----         ----          2025.02.01 │
└─────────────────────────────────┘
```

- 3×3 网格，每格 60–70px 圆角矩形
- 已达成：彩色 + 印泥质感（现有 SVG noise texture）+ 日期 + 微旋转（-2~2deg）
- 未达成：灰色虚线轮廓 + 标题可见 + `----`
- 进入视口时逐个盖章动画（translateY + scale + opacity），每个延迟 100ms
- **不做**：解锁时震动、emoji 动画——在 CSS-only 约束下保持简洁

成就条件全部可从 `__DATA__` 中计算，不需要额外数据源。

---

### D. 趋势图（SVG 重构）

**来源**：Claude（Canvas → SVG）+ Codex（精简标注）

| 改什么 | 具体做法 |
|--------|---------|
| 渲染引擎 | Canvas → SVG，统一 CSS 风格语言 |
| 标注精简 | 采用 Codex 建议：删除 BEST/WORST 常驻浮标，改为点击/触摸点位后在图表下方固定区域显示 |
| 线条风格 | 单色主线 + 目标红色虚线，回本交叉点用金色标记。不做手绘描边风格（开发成本高且难控制质感） |
| 网格 | 更浅，Y 轴标签减少到 3 档（Codex），清洁度优先 |
| 动画 | stroke-dashoffset 路径绘制（IntersectionObserver 触发），2s，简洁有效 |
| 交互 | 移动端吸附点位（Codex），tooltip 固定在图表下方不遮挡线条 |

**不做**：黑胶唱片纹理、驿站图风格、粒子化虚线——这些是装饰噪声。

---

### E. 入园战报

**来源**：Codex（按月折叠 + 抽屉明细）+ Claude（卡片个性化 emoji）

- 默认显示最近 4 次，其余按月份折叠分组
- 点击卡片展开当日明细（高度过渡动画，不做抽屉从底部滑出——在单 HTML 里 overlay 管理复杂）
- 每张卡片增加心情 emoji（采用 Claude 方案：🤩😊😅💸）——这是低成本高回报的个性化
- BEST/WORST 保留但改为侧边色条（Codex）而非大贴纸，减少视觉干扰
- **不做**：横向滑动相册、叠卡拖拽、视差效果——这些全是翻车高发区

---

### F. 消费分类

**来源**：Codex（排名表优先）+ 当前版本简化

当前的双向条形图认知成本确实偏高（三份方案共识）。改为：

```
┌─────────────────────────────────┐
│  消费结构                        │
│                                  │
│  🎢 入园节省  ¥1,820  ████████████│
│  🍔 餐饮     ¥180    ███         │
│  🛍️ 商品     ¥120    ██          │
│  🅿️ 停车     ¥80     █           │
│                                  │
│  净值  +¥1,440                   │
└─────────────────────────────────┘
```

- 单向条形图 + emoji + 金额，一目了然
- 条形长度按比例，最长的占满宽度
- 底部一行净值总结
- 分类色降饱和（Codex），用同色系明度区分而非高饱和碰撞
- **不做**：甜甜圈、双半环、环形罗盘——在 320px 屏幕上都不如列表好使

## 五、总结：一句话

三份方案本质是三种性格——Claude 是有远见但管不住手的创意总监，Codex 是冷静但缺少激情的产品经理，Gemini 是脑洞大开但不管工程死活的概念设计师。最优解是 **Codex 的克制 + Claude 的成就系统 + Gemini 的社交文案**，然后把其他所有花哨的东西全部砍掉。
